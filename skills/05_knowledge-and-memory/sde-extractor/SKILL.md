# Self-Definition Extraction Prompt

> **사용 방법**: 이 파일 전체 내용을 복사해서 너의 로컬 CLI(Claude Code, Codex CLI, Aider 등)의 첫 메시지로 붙여넣어. 출력된 `self-definition.md`를 외부 하네스 설계 대화로 가져와.

---

## Context

나는 이 환경의 사용자이고, 현재 "가장 온전한 형태의 로컬 하네스"를 설계하는 프로젝트를 진행 중이다. 외부 AI와의 협업 대화에서 (2) **사용자 모델** 층위 작업이 진행 중이며, 그 작업에 필요한 "사용자 자기 정의 데이터"를 너(로컬 CLI의 AI)가 이 환경에 접근 가능한 데이터로부터 추출해주길 원한다.

외부 대화에서는 이 환경의 데이터에 접근할 수 없기 때문에, 너의 작업 결과(`self-definition.md`)가 외부로 전달되는 유일한 채널이다.

---

## Task

다음 5개 영역에 대해 **나(이 환경의 사용자)를 정의하는 구조화된 마크다운 파일**을 생성한다.

| 영역 | 다룰 내용 |
|------|----------|
| A. 정체성 (who) | 역할, 배경, 활동 영역, 직업, 외부 정체성 |
| B. 지식 (what you know) | 전문 영역, 익숙한 도구·방법론·도메인 지식 |
| C. 사고 방식 (how you think) | 의사결정 스타일, 추상↔구체 선호, 메타인지 패턴 |
| D. 작업 패턴 (how you work) | 작업 흐름, 시간대, 협업 방식, 도구 사용 패턴 |
| E. 선호 (preferences) | 출력 형식, 톤, 길이, 깊이 (부차적 영역) |

---

## Data Sources to Examine

다음 순서로 데이터를 검토하라. 각 단계에서 무엇을 발견했는지 메모하라.

1. **이전 대화 기록** — 너의 환경에서 접근 가능하다면, 사용자와의 과거 대화 검색
2. **홈 디렉토리 설정 파일** — `~/CLAUDE.md`, `~/README.md`, `~/.config/*`, `~/NOTES*`, 개인 노트 등
3. **하네스 디렉토리** — 하네스 레포 또는 유사 경로의 하네스 구조
4. **활성 프로젝트 레포들** — README, package.json, pyproject.toml, 기타 메타 파일
5. **Git 활동** — `git log --author=<사용자> --all` 같은 명령으로 커밋 메시지·빈도·시간대·작업 도메인 패턴 추출
6. **셸 설정·도구 사용 흔적** — `.zshrc`, `.bashrc`, `.gitconfig`, 자주 쓰는 alias·함수, MCP 설정
7. **최근 작업 디렉토리** — 어떤 종류의 파일/프로젝트를 다루고 있는지

위 소스에 접근 권한이 없거나 비어있으면 그 사실을 결과 파일의 "추출 메타 정보"에 명시하라.

---

## Extraction Rules (반드시 준수)

1. **확신도 표시** — 모든 항목에 마커를 붙여라:
   - `✓` — 데이터에서 직접 확인됨
   - `◐` — 패턴에서 강하게 추정됨 (어떤 패턴인지 짧게 메모)
   - `?` — 데이터 부족, 사용자에게 직접 물어봐야 함

2. **단일 작업 일반화 금지** — 한 번의 작업이나 한 프로젝트에서 보인 행동을 "사용자의 본질"로 일반화하지 마라. 여러 데이터 포인트에서 반복적으로 관찰될 때만 패턴으로 간주.

3. **모름을 숨기지 마라** — 데이터로 답할 수 없는 항목은 추측해서 채우지 말고 `?`로 표시한 뒤 마지막 단계에서 사용자에게 물어라.

4. **출력의 톤·형식 같은 표면적 디테일에 매몰되지 마라** — E 영역은 마지막으로, 짧게.

5. **단정적 인격 평가 금지** — "사용자는 완벽주의자다" 같은 평가 대신 "데이터 라이프사이클(축적·정리·삭제·신선도) 관련 명령을 자주 사용함" 같은 관찰 기술.

---

## Process

1. **데이터 수집** — 위 7개 소스를 순서대로 검토. 각 소스에서 무엇을 발견했는지 한 줄씩 메모.
2. **초안 작성** — 5개 영역에 항목 채우기. 각 항목에 확신도 마커 부착.
3. **빠진 정보 정리** — `?` 표시된 항목들을 한 곳에 모은다.
4. **사용자에게 한 번에 질문** — `?` 항목들에 대해 한 번의 메시지로 모아서 물어라. 너무 많으면 가장 중요한 것부터 5개 이내로 추려라.
5. **답변 통합** — 사용자의 답변으로 `?` 항목을 채워라.
6. **최종 출력** — 아래 포맷대로 `self-definition.md` 파일로 저장.

---

## Output Format

다음 구조를 **정확히** 따라라. 외부 하네스 작업에서 자동 통합되어야 하므로 구조 변경 금지.

```markdown
# User Self-Definition

> Generated: YYYY-MM-DD
> Source Environment: [CLI 도구 이름]
> Extraction Sources Used: [실제 검토한 소스 리스트]

## A. 정체성 (who)
| 항목 | 내용 | 확신도 |
|------|------|--------|
| ... | ... | ✓/◐/? |

## B. 지식 (what you know)
| 항목 | 내용 | 확신도 |
|------|------|--------|

## C. 사고 방식 (how you think)
| 항목 | 내용 | 확신도 |
|------|------|--------|

## D. 작업 패턴 (how you work)
| 항목 | 내용 | 확신도 |
|------|------|--------|

## E. 선호 (preferences)
| 항목 | 내용 | 확신도 |
|------|------|--------|

## 추출 메타 정보
- 검토된 데이터 소스: [실제 접근한 것들]
- 접근 불가 소스: [있다면 명시]
- 확신도 분포: ✓ N개 / ◐ M개 / ? K개
- 사용자에게 직접 확인 받은 항목: [리스트]
- 관찰된 흥미로운 패턴 중 5개 영역 어디에도 안 들어간 것: [있다면 짧게]
```

## Anti-patterns (절대 하지 말 것)

- ❌ 사용자에게 5개 영역을 통째로 입력하라고 요구 — 그건 자기 선언이지 데이터 추출이 아니다
- ❌ 추측을 사실처럼 적기
- ❌ MBTI/성격 유형 같은 외부 프레임으로 분류하기
- ❌ 출력 톤·이모지 사용 같은 표면적 항목으로 표 채우기
- ❌ 한 번의 깃 커밋, 한 번의 대화에서 본 행동을 일반화하기

## Start

준비됐으면 시작해라. 첫 메시지에서는:
1. 어떤 데이터 소스에 접근 가능한지 한 줄씩 보고
2. 추출 작업 시작 — 진행 상황을 단계별로 보여주며

진행 중 막히면 즉시 멈추고 사용자에게 물어라. 추측 금지.
