01

전체 노선도

본선(S1→S8)을 따라 진행하며, 마름모(◆)는 인간 게이트입니다 — G1·G3은 루프가 스스로 통과할 수 없는 정지점, G2는 v1.4부터 3단 처리(위임 실행 / 큐 적재 후 속행 / 선행 조건일 때만 정지)로 야간 무인을 유지합니다. 붉은 지선은 검수 FAIL 시 재실행 루프(최대 3시도), 점선 위 우회로는 조건부 시장조사, 청록 점선은 codex(GPT-5.5) 이종 검수 접점입니다.

읽는 법 — 본선: phase 전이(state 파일 즉시 갱신) / ◆: 인간만 통과시킬 수 있는 역 / 붉은 지선: 최대 3바퀴 / 점선 레일(하단): 턴이 끊겨도 훅이 본선으로 되돌려 놓음

03

인간 게이트 3곳 — 절대 우회 금지

게이트 phase(awaiting_*)에서는 드라이버 훅도 개입하지 않습니다 — 루프는 자연 정지하고, 인간의 다음 발화만이 재개시킵니다. v1.4부터 G2는 기본 비정지(위임·큐) — 정지형 게이트는 G1·G3과 G2의 선행 조건 분기뿐입니다.

G1계획 승인
  • 위치: S4 말 — ExitPlanMode 호출 자체가 게이트
  • 인간이 보는 것: 계획 + 자가검토 5관점 + critic 비평 + GPT-5.5 교차검토
  • 승인 → execute / 거부·수정 → plan 회귀
  • AC는 승인 시점에 동결 — 이후 변경 금지
G2외부 영향 — 3단 처리v1.4
  • 위임됨(게이트 1에서 승인된 pre_approved_external): 즉시 실행 + operations.log 자기 증명
  • 미위임·비선행: 큐(pending_external_actions) 적재 → 루프는 멈추지 않고 속행, S8 후 일괄 결정
  • 미위임·선행 조건: 그때만 awaiting_user 정지
  • 훅 강제: PreToolUse가 push·발송류를 도구 레벨 차단 — 규율 아닌 메커니즘. 자동 실행 금지 원칙 불변
G3재시도 한도
  • 위치: S7 — iteration 3 소진 시
  • 보고 의무: 미해결 AC 목록 + 시도별 실패 원인 + 5모드 분포
  • 3택 제시: 부분 수용 / 중단 / 직접 지시
  • 성공 위장 절대 금지 — 정직 보고가 합격보다 우선

케이스별 경로

실제 루프가 밟을 수 있는 8가지 대표 경로입니다. 칩의 색이 행위 주체를 나타냅니다 — 범례는 문서 상단과 동일합니다.

A표준 합격 — 시도 1에 끝나는 해피 패스조건: 명세형 지시 · 검수 전원 PASS
S1 구체화패스스루 (질문 0~2)→ S2 skip외부 사실 불요→ S3 계획+AC 고정→ S4 critic OKAY→ codex 교차검토→ G1 승인→ S5 실행 (opus)→ S6 — 3검수 전원 PASS→ S8 적재→ done · state 삭제

e2e 1회차(프로젝트 예시 디자인)·3회차(하네스 비교 HTML)가 이 경로 — 시도 1, AC 전 항목 PASS.

B검수 FAIL → 차분 재실행 → 합격조건: FAIL 존재 ∧ iteration < 3
S5 실행 (시도 1)→ S6 — AC-3 FAILspec PASS · verify FAIL · codex FAIL→ S7 차분 피드백{AC-3, 검수자, 증거, 기대vs실제, 수정지시}→ S5 재실행 (시도 2)FAIL 항목만 수정 — 전체 재작업 금지→ S6 재검수 — 전원 PASS→ S8→ done

codex가 발견한 FAIL도 검수자 출처 표기로 같은 피드백에 합산됩니다(v1.3) — executor는 어느 검수자가 무엇을 지적했는지 보고 수정합니다. e2e 2회차가 이 경로(픽셀 검수가 캡처 결함 적발 → 시도 2 합격).

C한도 소진 — 게이트 3 정직 보고조건: 시도 3까지 FAIL 잔존
시도 1 FAIL→ 시도 2 FAIL→ 시도 3 FAIL→ G3 한도 소진→ awaiting_user미해결 AC + 시도별 원인 + 5모드 분포→ 3택: 부분 수용 / 중단 / 직접 지시

성공 위장 절대 금지 — 어떤 선택이든 S8 적재(실패 교훈 포함)는 수행됩니다.

D계획이 통과하지 못할 때 — critic REJECT / 인간 거부위치: S4
D-1 · loop-critic REJECT (자동 수정 루프)
S3 계획→ critic REJECTtop 3~5 개선안→ 계획 차분 수정→ 재비평 (최대 2회)→ OKAY → 게이트 1로/ 2회 후에도 REJECT → awaiting_user
D-2 · 게이트 1에서 인간 거부
G1 검토자가검토+critic+GPT 의견 열람→ 거부·수정 지시→ S3 회귀 (plan)계획 재작성 후 다시 S4

critic 루프는 무한 비평을 막기 위해 2회 상한 — 기계끼리 합의 못 하면 인간에게 올립니다.

E외부 영향 — 게이트 2 3단 처리위치: S5 실행 중 (v1.4)
E-1 · 위임됨 — 무정지 (게이트 1에서 사전 승인)
S5: git push 필요→ pre_approved_external 매칭훅 통과→ 즉시 실행 + 로그→ 루프 속행
E-2 · 미위임 + 비선행 — 큐 적재 후 속행 (v1.4 기본)

E-3 · 미위임 + 선행 조건 — 그때만 정지
push 없인 후속 AC 불가→ G2→ awaiting_user 정지→ 승인 → 위임 목록 추가 → 재시도(훅 통과)

자동 실행 금지 원칙은 불변(14절 1순위) — v1.4가 바꾼 것은 정지 범위(전체→해당 작업만)와 강제 계층(자진 신고 규율→PreToolUse 훅)입니다.

Fcodex 검수의 4분기위치: S6 (v1.3 자문 모드 파일럿)
F-1 · 자문 FAIL — 비차단, 피드백 합산

GCritical Path — 강화 메커니즘 총동원조건: 외부 영향·자동화 연계·복구 비용·명시 "중요" 중 1+

N=3 판정 분포

tier-light 강등 금지 — Critical에서 N=3은 불변입니다. codex 실패 시 Claude 3종으로 폴백해 N=3을 유지합니다. 답변 끝 5모드 요약 의무.

H중단·재개 — 훅 한도, 세션 끊김, 수동 탈출운영 한계 상황
H-1 · 같은 phase 2회 차단 / 누적 12회 → 정직 정지
Stop 훅 차단 k=2 (동일 phase)→ stalled무한루프 방지 — 수동 개입 보고 + 5모드
H-2 · 세션 재시작·컴팩션 → state 기반 재개
세션 끊김 / 컴팩션→ 사용자 "루프 재개" 발화→ state 읽기 → session_id 갱신→ 해당 phase부터 속행단계 전이 flush 덕분에 무손실
H-3 · 수동 탈출
사용자 "루프 중지"→ aborted 기록 + state 삭제/ 폴백: rm .claude/full-loop-state.local.json

state 파일이 손상되면 훅은 무반응(fail-open) — 루프를 억지로 살리지 않고 스킬 재생성 또는 사용자 보고로 넘깁니다. TTL 24h 경과 state는 훅이 무시합니다.

드라이버 훅 & 상태 파일 — 루프가 죽지 않는 이유
