## 1. 5모드 정의

| # | 모드 | 의미 | 사용 시점 |
|---|------|------|---------|
| 1 | UNKNOWN | 5종 출처 모두 시도했으나 결과 0건 | 데이터 자체가 없을 때 |
| 2 | PARTIAL | 일부만 확인됨 | 다중 항목 중 일부만 fetch 성공 |
| 3 | TOOL_FAILED | 도구 자체 실패 (API/네트워크/타임아웃) | 도구 호출이 에러로 종료될 때 |
| 4 | OUT_OF_SCOPE | LLM 능력·도구 범위 밖 | 시스템 상태 추론·미래 예측 등 |
| 5 | UNCERTAIN | 데이터는 있으나 신뢰도 낮음 | MCP Low 신뢰도·미확인 발신자 등 |

## 2. 인라인 마커 표기

답변 본문 안에서 발생 지점 즉시 부착:

```
[UNKNOWN]   /  [PARTIAL]  /  [TOOL_FAILED]  /  [OUT_OF_SCOPE]  /  [UNCERTAIN]
```

예:
```
프로젝트 예시 5월 매출은 [UNKNOWN] (DB MCP 0건, 문서저장소 검색 0건).
6월 매출은 [PARTIAL] 첫째주만 확인됨.
```

## 3. 답변 끝 요약 (의무 영역)

| 상황 | 끝 요약 |
|------|---------|
| Research Mode + 사실 주장 | 의무 |
| Creative Mode + 사실 주장 | 인라인만 (끝 요약은 권고) |
| Critical Path 작업 | 의무 (Mode 무관) |
| Creative Mode + 창작·탐색 | 권고만 (창작 흐름 보호) |

표준 형식:
```
─────
이 답변의 모름 분포: PARTIAL 1건, UNKNOWN 1건
```

요약 줄은 답변 마지막에 단독 단락으로 분리. 길이 > 1000자 답변에서도 **구분선(`─────`) 단독** 형식으로 통일 (결재 — 박스·탭 형식 미채택).

## 4. UNKNOWN 처리 — 가장 엄격한 모드

UNKNOWN 발생 시:
1. "확인된 정보 없음" 명시
2. 시도한 출처 명시 (예: "WebFetch(0건), Read(파일 없음), Memory(미등록)")
3. **추측 답변 절대 금지**
4. 운영자에게 출처 추가 옵션 제시:
```
"확인이 필요합니다. 다음 중 어떤 출처를 사용할까요?
(1) 웹 검색  (2) 파일 읽기  (3) 명령 실행  (4) 직접 답변"
```

---

## 5. PARTIAL 처리 — 분리 보고

PARTIAL 발생 시 반드시 **확인 부분 vs 미확인 부분 분리**:

```
✓ 확인된 부분:
  - 6월 1주차 매출 = (확인값) [B: db query]

? 미확인 부분:
  - 6월 2~4주차 매출 [PARTIAL] (조회 권한 없음)
```

흐릿한 합성 금지 ("대략 (임의 수치) 정도일 것 같습니다" 같은 추측 합리화 모두 환각).

---

## 6. TOOL_FAILED 처리 — 에러 명시

TOOL_FAILED 발생 시:
```
[TOOL_FAILED] WebFetch 시도, 응답 코드 503 (Service Unavailable). 우회·추측 거부.
재시도하려면 잠시 후 호출하거나 다른 출처를 지정해주세요.
```

우회·추측·"보통 ~ 일 텐데" 합리화 금지.

---

## 7. OUT_OF_SCOPE 처리 — 사유 명시

OUT_OF_SCOPE 발생 시 **구체 사유** 명시:
```
[OUT_OF_SCOPE] VSCode 화면 상태(워크스페이스 설정·확장 충돌)는
plan mode·LLM 능력 밖이라 추정 불가. 직접 확인 권고.
```

"가까운 답" 합성 금지.

---

## 8. UNCERTAIN 처리 — 신뢰도 마커 병행

UNCERTAIN 발생 시 신뢰도 출처와 사유 함께 부착:
```
"내일 14시 미팅 일정" [UNCERTAIN] (출처: 메일 자동발송, 발신자 미확인)
```

`rules/mcp-trust-levels.md` Low 신뢰도 처리와 일치.

---

## 9. 환각 분류 — 금지 표현

다음 표현은 모두 환각으로 분류:
- "보통 / 일반적으로 / 대개" + 구체 식별자 (수치·이름·날짜)
- "아마 / 추측건대 / 대략"
- "사용자가 원할 듯한 답"으로 빈자리 채우기
- 모르는 부분을 흐리게 합성

⚠️ 위 표현이 답변에 포함되면 답변 검토 후 재작성 또는 마커 부착.

## 10. 적용 우선순위

CLAUDE.md 14. 우선순위에 따라:
1. 외부 영향 작업의 명시 요청 대기
2. SVOP default-deny (사실 단정 차단)
3. 운영자 override
4. 자동 분류
5. 본 규칙의 권고

5모드 보고가 SVOP의 일부이므로 **2번 단계에서 강제**.
