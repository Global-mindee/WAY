## 0. 프로젝트 개요

이 설계도는 한 1인 운영자의 실제 사례에서 출발했습니다. 고유한 개인 데이터는 모두 일반화되었으며, 온보딩이 당신의 맥락으로 재해석합니다.

### 목적
로컬 CLI 환경에서 적용할 **가장 온전한 형태의 하네스**를 설계한다.
사용자의 추상적 개념을 구체화하여, 어떤 작업이 들어와도 동일하게 작동하는
**얇고 강건한 운영 원칙**을 정립한다.

### 범위 — 하네스 vs 프로젝트 경계
- **하네스**: 작업 환경, 원칙, 도구 골격. 거의 변하지 않는다.
- **프로젝트**: 별도 레포에서 운영. 변동성을 흡수하는 곳.
- 하네스는 프로젝트들을 **가로지르는 불변의 OS**다.

### 작업 방식
- 4개 층위를 순차적으로 심화: (1) 철학 → (2) 사용자 모델 → (3) 신뢰성 → (4) 운영
- 각 층위 완료 시 본 문서에 누적 반영
- 미해결 난제와 리서치 필요 항목은 명시적으로 파킹

## (1) 철학 (왜 존재하는가)

### 핵심 명제
> **규칙은 간단명료(심플)할수록 강하다. 단, 그 규칙은 어떤 상황에서도 깨지지 않아야 한다.**

수많은 규칙과 제약사항은 결국 사용자가 새롭게 얻을 수 있는 지식까지 모두 제약한다.
근본 원칙 몇 개로 변동성을 흡수하는 구조가 모든 작업 유형(리서치·분석·기획·실험·운영)에 대응하는 유일한 방법이다.

---

### 원칙 (Principles)

#### P1. Plan-First
실행 전에 사용자가 원하는 결과가 구체적으로 정의되지 않으면 어떤 작업도 시작하지 않는다.
추상적 아이디어를 데이터화·구체화하는 단계가 모든 실행에 선행한다.

#### P2. Sync-Before-Execute
사용자와 AI의 생각이 **90% 이상 일치**해야 실행 단계로 넘어간다.
이를 위해 토론·논의·질문 단계를 의도적으로 둔다.
싱크가 부족한 상태의 실행은 시간 낭비를 넘어 잘못된 신뢰를 만든다.

#### P3. Plan-Execute Separation
**계획 단계는 인간 주도, 실행 단계는 AI 주도.** 두 단계를 섞지 않는다.
명확한 계획 혹은 PRD가 완성된 후의 실행은 모두 AI에게 위임할 수 있다.

#### P4. Immutability (불변성)
하네스의 구조와 원칙은 환경이 바뀌어도(노트북·서버·다른 OS) 동일하게 유지된다.
하네스 수정은 늘 조심스럽게 다룬다.
이 원칙은 P6의 메타 피드백 메커니즘과 결합되어 **즉흥적 수정 금지**로 작동한다.

#### P5. Data-Driven Dependency Criterion
**외부 의존을 받아들이는 기준은 "새로운 데이터/관점을 가져오는가"이다.**
- 데이터·관점의 통로(MCP, 데이터베이스, 외부 지식) → 받아들임
- 기능을 흉내내지만 얇은 프롬프트 래퍼인 의존(상당수 플러그인·에이전트·스킬) → 배제
- 진짜 전문가급 능력은 도구의 양이 아니라 도구가 접근하는 데이터의 질에서 나온다

#### P6. Operational One-Way + Meta Feedback Loop
**데이터는 새지 않되, 교훈은 수집된다 (Data Doesn't Leak, Lessons Do).**
- **운영 데이터**(프로젝트 컨텍스트·산출물·작업 내용)는 프로젝트 → 하네스로 역류하지 않는다
- **메타 피드백**(하네스 작동 양상·마찰 지점·반복 패턴)은 별도 채널로 수집되어 하네스 개선의 근거가 된다
- 하네스 수정은 누적된 메타 피드백에 근거할 때만 허용된다 (P4 강화)

---

### 안티원칙 (Anti-Principles)

> 명시적으로 **채택하지 않는** 원칙들. 무엇을 원칙이 아니라고 선언하는 것은 무엇을 원칙으로 두는 것만큼 중요하다.

#### AP1. 비결정성은 자산이다 (Non-determinism as a Feature)
동일 입력에도 컨텍스트·시간·축적 데이터·관점·스킬 조합에 따라 결과가 미묘하게 달라지는 것은 **정상이며 바람직하다**.
일관성/재현성을 위해 이 변동성을 억제하지 않는다.
AI는 수학 문제를 풀어주는 계산기가 아니다.

#### AP2. 명시성은 원칙이 아니다 (Explicitness is Not a Principle)
모든 출력의 추적가능성/감사가능성을 원칙으로 두지 않는다.
이유:
- 명시성은 할루시네이션을 *줄이는* 것이 아니라 *발견하기 쉽게* 만들 뿐
- 검증 책임을 사용자에게 떠넘기는 구조라 사용자의 시간·집중력에 따라 결과 품질이 크게 달라짐
- "AI를 협업 파트너로" 보는 정신과 충돌 (AI를 의심 대상으로 두고 매번 감사하는 구조가 됨)

단, (4) 운영 층위에서 "**기록/로그**" 수준의 가벼운 추적은 도입한다 — 이는 P6의 메타 피드백 수집을 위한 보조 장치이지, 사용자 검증을 위한 장치가 아니다.

#### AP3. 데이터는 능동적으로 격리되지 않는다 (No Active Data Quarantine)
시간 기반·신호 기반 자동 stale 검출은 하지 않는다. 데이터의 부재나 충돌은 stale 신호로 간주하지 않는다.
- 새 데이터가 기존 데이터에 도전할 때만 Tier 이동·신뢰도 변경이 일어난다 (Push 기반).
- 100년 진리·가끔 쓰는 불변 지식·드물게 쓰지만 핵심인 데이터는 능동적 약화로 잃지 않는다.
- AI는 분석·통계만 제공하고, 변경 결정은 사용자가 한다 (능동적 변경 권고 안 함).

이 안티원칙은 (2) Item 2(메모리 라이프사이클), (3) Item 5(MCP 신뢰도), (4) 영역 C·D(신뢰도 변경·메타 피드백 임계치)의 토대로 작동하므로, v0.6에서 (1) 안티원칙으로 격상되었다.

## (2) 사용자 모델 (누구를 위한 것인가)

### 핵심 명제
> **사용자 모델은 단순 프로필이 아니라, P1·P2·P3의 트리거 조건이다.**

- 저장소가 아니라 **분류기** (이 입력을 어떤 모드로 처리할지 결정)
- 개인화는 출력의 톤이 아니라 **상호작용의 흐름 자체를 바꾼다**
- 분기는 환경이 아니라 **입력 유형·맥락**에서 일어나야 함 (환경별 분기는 P4 위반, 통합 대상)

### 3개 Sub-Question 구조
1. **무엇을** 알아야 하나 (what) — sub-q 1 ✓ 완료
2. **어떻게** 알게 되나 (how to acquire) — sub-q 2 ✓ 완료
3. **어떻게** 진화하나 (how it evolves) — sub-q 3 진행 예정

**5개 영역 분류**:

|------|------|------|

### Sub-Q 2 결과 — 어떻게 알게 되나 ✓

**모듈명**: Self-Definition Extractor (SDE)

**실행 주체**: 신설 하네스 내부 (모든 환경에서 동일하게 작동 — P4 준수)

**트리거 (4계층)**:
- **초기**: 하네스 첫 설치 시 자동 실행
- **정기**: 월 1회 (30일) — 사용자 결정
- **이벤트**: 환경 추가, 사업 우선순위 변경 명시, 충돌 데이터 감지 시
- **명시**: 사용자 "갱신" 요청 시
- ※ AI 자동 감지 트리거는 채택 안 함 (AP2 명시성 원칙 거부와 정합)

**데이터 출처 (4 Layer)**:
- **Layer 1**: 로컬 파일시스템 + Git
- **Layer 2**: 메모리 디렉토리 + 대화 기록
- **Layer 3**: MCP 외부 시스템 전면 활용 — 사용자 결정 (Calendar·Gmail·Drive·Notion 등)
- **Layer 4**: 사용자 직접 답변 (Layer 1-3에서 ? 채울 때만)

**검증 (사용자 작업 패턴 흡수)**:
- SVOP default-deny — 5종 출처(USER·READ·BASH·WEB·MEMORY) 외 사실 단정 차단
- 확신도 마커 (✓/◐/?) 의무
- sample_size ≥ 3 또는 [3/3] 마커
- 모름 보고 4모드 (UNKNOWN/PARTIAL/TOOL_FAILED/OUT_OF_SCOPE)

**산출물**:
- 동일 포맷의 통합 `self-definition.md`
- 이전 버전과의 diff (→ sub-q 3 영역)
- 추출 메타 정보 (출처·접근 불가·확신도 분포)

**미해결 항목 (다른 층위로 이월)**:
- Layer 3 MCP 데이터 출처별 검증 정책 → (3) 신뢰성 층위
- 월 1회 트리거의 cron 구현 / P6 회로 연동 → (4) 운영 층위

### Sub-Q 3 결과 — 어떻게 진화하나 ✓

5개 Item으로 구성. 각 Item의 핵심 결정 사항만 정리.

#### Item 1: 변화 추적 (Change Detection)

- **변화 유형 6종**: 신규 / 갱신 / 강화 / 약화 / 이동(Archive) / 충돌
- **Critical 임계치 — "가장 넓게"**: 충돌 + C·D 핵심 항목 갱신 + ★ 답변 모순 + A·B 영역 변경 모두 포함
  - 선택 근거: "자동화로 연계되기 위해서는 초기 데이터의 신선도와 신뢰성이 중요"
- **Background 로그**: `background-log.md` 별도 파일 (never_delete 충족 + 메인 changelog 노이즈 방지)
- **산출물 5종**:
  - `self-definition.md` — 최신 통합본
  - `self-definition-changelog.md` — Critical·Notable 변화 누적 로그
  - `self-definition-archive/` — Archive Tier로 이동한 항목 (never_delete)
  - `self-definition-conflicts.md` — 미해결 Conflict 항목 (사용자 결재 대기)
  - `background-log.md` — Background 등급 변경 누적

#### Item 2: 메모리 라이프사이클

- **적용 범위**: 신설 하네스 전체 메모리 (SDE 데이터에만 한정 X)
- **4타입 분류** (사용자 기존 패턴 흡수): User / Project / Feedback / Reference
  - P6 정합: Project 카테고리는 프로젝트 메타 정보만, 운영 데이터는 외부 레포 격리
- **3-Tier 구조**: Public(활성) / Quarantine(격리) / Archive(보존, `never_delete = true`)
- **신선도 정책 — Push 기반** (사용자 통찰):
  - 능동적 stale 검출 없음 (시간·신호 기반 자동 격리 없음)
  - 새 데이터가 기존 데이터에 도전할 때만 Tier 이동
  - "데이터의 부재"는 stale 신호로 간주 안 함 (100년 진리·가끔 쓰는 불변 지식 보호)
- **데이터 부재 처리** (옵션 3 + 사용자 확인):
  - 자동 약화 + Quarantine 이동 안 함
  - Quarantine 후보로 등록 → Critical 결재 요청 (Item 1과 통합)
  - 사용자 선택지: 여전히 유효 / 검증 필요 / 폐기
- **접근 정책**: Public → 자동 인젝션 + 인덱스 검색 / Quarantine·Archive → 명시 호출만

#### Item 3: 의도-환경 동기화

- **두 상태의 명시적 분리**: self-definition.md 안에 영역(A~E)마다 별도 섹션
  - 선언적 상태 (Declared) — 사용자 명시 (★)
  - 관찰적 상태 (Observed) — SDE가 환경에서 추출
  - Gap 추적 — 둘 사이 차이
- **Gap 분류 3종**:
  - 활성 전환 (Active Transition) — 의도된 변화 진행 중
  - 정체 (Stagnation) — 선언만 있고 행동 변화 없음
  - 모순 (Contradiction) — 선언과 행동이 정반대
- **자동 분류 정책 — 하이브리드**:
  - 모순(Contradiction)만 시스템 자동 분류 + Critical 결재 (Item 1과 통합)
  - 활성 전환 / 정체는 사용자가 직접 분류

#### Item 4: P6(운영 단방향 + 메타 피드백) 회로와의 관계

- **채널 구조 — 단일 메타 피드백 채널 + 출처 태그**:
  - Layer 분리 없음 (회색 영역 분류 비용 회피)
  - 출처 태그: `[작동 관찰]` `[SDE 추출]` `[사용자 명시]` `[Gap 추적]` (복수 부착 가능)
- **상호 갱신 회로 불필요**: 같은 채널 안에서 자연스럽게 처리 (별도 회로 설계 안 함)
- (1) 핵심 명제 "간단명료" 정합 + 사용자의 `lessons → wiki → intents` 폐쇄 회로 패턴과 정합

#### Item 5: 변화의 결재 흐름

- **결재 요청 형식 — 묶음 보고**: 월 1회 SDE 갱신 직후, 모든 Critical 항목을 `pending-decisions.md`로 통합 노출
- **결재 응답 형식**: 체크박스 + 직접 파일 입력 (사용자가 해당 파일에 결정 직접 기록)
- **대기 중 self-definition.md 정책**: 결재 전 기존 파일 유지, 결재 후 적용 (P2 Sync-Before-Execute 정합)
- **미응답 시 정책 — 무기한 대기**: 자동 만료 없음, 알림 강화 없음
- **알림 3계층** (설계상 구조 — 각 계층의 실제 구현 상태는 `CLAUDE.md` 9절 참조):
  - **계층 1 (기본)**: 하네스 작업 시작 시 `pending-decisions.md` 자동 노출

## (3) 신뢰성 (어떻게 믿을 수 있는가)

### 핵심 명제
**Prevention over Verification (검증보다 차단)** — LLM 아키텍처상 할루시네이션의 근본 해결은 불가능하다는 전제 위에서, "LLM에게 추측할 자유를 구조적으로 주지 않는다"는 패러다임을 채택.

### 채택된 골격: 시나리오 5 (Anthropic Native)
2026-05-23, 데스크 리서치 v2.0(`research-hallucination-mitigation.md`)을 토대로 채택. Anthropic 자체가 Claude 모델을 "Refusal over Guessing"으로 calibrate한다는 발견과 정합. 1인 운영 환경 최적화.

**시나리오 5 구성**:
```
[기본 prompting — 모든 작업]
  - "I don't know" 명시적 허용
  - External knowledge restriction (요청 시)
  - Citations 의무화 (사실 작업 한정)

[Mode Toggle]
  - Research Mode → 기본 + CoT verification + Direct quotes
  - Creative Mode → 기본만 (창의 작업 보호)

[Critical Path 한정]
  - Best-of-N verification (N=3)
  - 의무 외부 데이터 조회(fetch)
```

### Sub-Q 결과 — 5개 Item

#### Item 1: Mode Toggle (사실 진술 vs 추론·창작의 분리)

- **모드 명칭**: Research Mode / Creative Mode (Default → Creative 변경, 사용자 결정)
- **모드 결정 주체**: 하이브리드 — AI 자동 분류 + 1줄 override
- **작업 유형 분류**: 3종 — 사실 진술 우선 / 창작·탐색 우선 / 혼합(회색 영역)
- **회색 영역 정책**: 보수적 — 회색 영역은 Research Mode 기본
- **모드 활성화 시점**: 자연 전환 + 사용자 알림

#### Item 2: 외부 데이터 조회(fetch) 의무화

- **표기 정책**: 외래어는 "한국어 풀이(영어)" 형태 (사용자 결정 — 예: 외부 데이터 조회(fetch), 트리거(trigger))
- **fetch 의무화 트리거**: Mode 무관 — 사실 주장은 Research·Creative 모두 동일 의무
- **출처 우선순위**: USER → BASH → READ → MCP → WEB → MEMORY (SVOP 5종 흡수 + MCP 추가)
- **fetch 실패 시**: "I don't know" 발동 + 모름 4모드 보고 (UNKNOWN/PARTIAL/TOOL_FAILED/OUT_OF_SCOPE)

#### Item 3: 모름을 강제로 표면화 (카파시 #1 흡수)

- **모름의 분류 체계**: 5모드 — 기존 4모드 + UNCERTAIN 보강 ("데이터는 있으나 신뢰도 낮음")
- **모름의 노출 위치**: 인라인 마커 + 답변 끝 5모드 요약 (병행)
- **적용 영역**: Research Mode / Critical Path / 사실 주장 → 의무. Creative Mode + 창작·탐색 → 권고만 (강제 안 함)
- **표준 응답 형식**: 인라인 `[UNKNOWN]/[PARTIAL]/[TOOL_FAILED]/[OUT_OF_SCOPE]/[UNCERTAIN]` 마커 부착 + 답변 끝 모드 분포 요약

#### Item 4: 위험도 분류 및 Critical Path 방어

- **Critical 정의 — 본 항목 vs (2) Item 1 구분**: 본 항목의 Critical = **작업 자체의 위험도** / (2) Item 1의 Critical = 자기 정의 데이터 변경의 위험도 (서로 다른 차원)
- **Critical Path 4기준** (하나라도 해당하면 Critical):
  - 외부 영향 (External Impact)
  - 자동화 연계 (Automation Downstream)
  - 되돌리기 비용 (Reversibility Cost)
  - 명시 표시 (사용자가 "중요" 표시)
- **분류 체계**: 이진 분류 (Critical / Non-Critical)
- **Critical 방어 메커니즘**: Best-of-N (N=3) + 강제 Research Mode + 의무 fetch + 답변 끝 5모드 요약
- **Best-of-N 결과 처리**:
  - 3/3 일치 → 답변 채택 + Citation
  - 2/1 일치 → 답변 채택 + 불일치 1건 `[DISAGREE]` 마커
  - 분산 → 답변 보류 + 사용자 검토 요청

#### Item 5: Layer 3 MCP 출처별 검증

- **신뢰도 등급 체계**: 3단계 (High / Medium / Low)
- **출처별 기본 매핑**:
  - **High**: Supabase 전체 / Calendar·Drive 본인 직접 입력·작성 / Gmail 본인 발신
  - **Medium**: Calendar·Drive 공유받은 항목 / Kiwi 항공권 API
  - **Low**: Gmail 받은편지(모르는 발신자) / Gmail 스팸 / 자동 발송 메시지
- **Low 신뢰도 처리**: 답변 시 `[신뢰도: Low]` 마커 의무 부착 (Item 3과 정합)
- **충돌 시 처리**: 두 출처 모두 명시 + PARTIAL 매핑 → Critical 작업이면 Best-of-N 흐름 자동 진입

### 영역 B: 자동화·트리거·알림 (Automation)

> 본 영역의 cron·자동 commit/push·대시보드는 모두 **설계상 구조**다. 실제 구현·가동 상태는 `CLAUDE.md` 9절·13절·부록 B를 참조한다.

#### cron 실행 환경
- **상시 가동 머신([작업 머신 B])** + 새벽 03:00 (운영자 야간 작업 시간대를 피해)

#### cron 흐름 (3종, 영역 D 보강 포함)
| 빈도 | 시각 | 작업 |
|------|------|------|
| 매일 | 03:00 | 결재 미응답 3일 체크 + Gmail 알림 |
| 매월 1일 | 03:00 | SDE 트리거 + 메타 피드백 집계 + 임계치 분석 |
| 매분기 첫 날 | 03:00 | 분기 리뷰 보고서 + 결재 요청 자동 등록 |

#### 알림 3계층 구체 구현
- **계층 1 (기본)**: `CLAUDE.md`에 지시 추가 → 작업 시작 시 `decisions/pending.md` 자동 확인 → 대기 항목 있으면 AI 첫 응답에 노출
- **계층 2 (보조)**: `operations/dashboard` (localhost:4173)에 결재 큐 카운터 + 미응답 일수 표시
- **계층 3 (안전망)**: cron 매일 03:00 → 3일 경과 항목 있으면 Gmail MCP로 1회 발송 (항목별 1회만, 반복 안 함)

#### 자동 commit·push 구체 메커니즘
- **자동 commit**: 5분 단위 묶음. append-only는 `[auto] {파일명} updated`, 일반 영역은 AI가 1줄 요약 + `[auto]` 접두어
- **자동 push**: 1시간 단위 묶음 (네트워크·Git 서버 부하 완화)
- **Conflict 처리**: 자동 merge 시도 → conflict 시 자동 push 중단 + `pending.md`에 conflict 항목 등록 + 알림 계층 1·2 발동. 매뉴얼 해결

#### 이벤트 기반 자동화 트리거
영역 A의 이벤트 6종을 자동화 흐름에 매핑. 자기 정의 Critical 변경 → `pending.md` 자동 등록, MCP 충돌 + Critical 작업이면 Best-of-N 자동 진입, 결재 미응답 3일 → Gmail 알림 등.

### 영역 C: (3) 신뢰성의 운영 정책 (Reliability Ops)

#### 신뢰도 변경 정책 — 정적 + 명시 변경 (AP3 정합)
- 한 번 정해진 신뢰도는 능동적 자동 약화 안 함
- 변경은 사용자 명시 결정만
- 충돌 누적은 P6 메타 피드백(영역 D)에서 통계 제공, 자동 약화 없음

#### MCP 추가·삭제 시 신뢰도 자동 매핑
- 신규 MCP 연결 → AI 자동 매핑 시도 → `pending.md` 결재 요청 자동 등록 → 사용자 결재 후 `rules/mcp-trust-levels.md`에 반영
- MCP 해제 시 매핑 정보는 `archive/`로 이동 (never_delete)

#### 신뢰도 마커 시각적 표현 — 출처 인용에 통합
- High: 표기 생략 가능 (예: `(출처: Supabase)`)
- Medium: 표기 의무 (예: `(출처: Drive 공유 문서, 신뢰도: Medium)`)
- Low: 표기 의무 + 사유 명시 (예: `(출처: Gmail, 신뢰도: Low — 발신자 미확인)`)

#### Mode Toggle 분류 정확도 측정 — override만 기록
- AI 자동 분류 → 사용자 명시 override 발생 시만 `logs/operations.log`에 기록
- 형식: `[timestamp] mode-override: AI=Research, User=Creative, 작업="{요약}"`
- 영역 A의 로그 이벤트 5종 → 6종으로 확장

### 영역 D: P6 메타 피드백 회로 (Meta Feedback Loop)

#### 수집 대상 — 5종 메타 피드백
1. **Mode override 누적** ([작동 관찰])
2. **MCP 출처 충돌 누적** ([작동 관찰])
3. **Best-of-N 결과 분포** (3/3 vs 2/1 vs 분산) ([작동 관찰])
4. **결재 미응답 패턴** ([작동 관찰] + [사용자 명시])
5. **자기 정의 변화 누적** (`changelog.md`에서 자동 추출) ([SDE 추출] / [사용자 명시])

(2) Item 4의 출처 태그 4종([작동 관찰] / [SDE 추출] / [사용자 명시] / [Gap 추적]) 부착 의무.

#### 저장 형태
- Raw 누적: `logs/meta-feedback.log` (append-only)
- 월 1회 집계: `logs/meta-feedback-summaries/{YYYY-MM}.md` (cron 자동 생성)

#### 임계치 기반 자동 결재 요청
임계치 초과 시 `pending.md`에 결재 요청 자동 등록. AI는 분석·통계만 제공, 권고안 자동 작성 안 함 (AP3 정합).

| 임계치 | 조건 |
|--------|------|
| Mode override 비율 > 30% | 특정 작업 유형에서 자동 분류 자주 틀림 |
| MCP 출처 충돌 > 5건/월 | 특정 출처 신뢰도 재검토 필요 |
| Best-of-N 분산 비율 > 10% | Critical 작업 신뢰성 저하 신호 |
| 결재 미응답 > 7일 (한 항목) | 결재 부담 증가 신호 |
| 자기 정의 동일 영역 충돌 ≥ 3회/월 | 그 영역 정의 자체 재검토 필요 |

#### 하네스 진화 3계층 흐름
